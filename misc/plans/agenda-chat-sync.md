# Agenda Chat Persistence + Cross-Client Sync

## Context (what exists today)

- UI chat (`apps/agenda/src/ui/app.js`) keeps `messages` only in-memory; it calls `GeistService.interpret(text)` and appends `{ role, content }` locally.
- CLI (`apps/agenda/bin/agenda.js`) calls `GeistService.interpret(text)` and prints the result; no shared history.
- `DatabaseService` already has Redis and a write-only Redis Stream (`eventStream = agenda:events`) via `publishEvent()`, but no consumer and no “chat” persistence.
- SMS is described in `misc/plans/agenda.md` (Twilio + HTTP port), but there’s no inbound/outbound SMS implementation in `apps/agenda/` yet.

Goal: one shared, persistent conversation timeline that all clients (UI/CLI/SMS) read/write, with real-time-ish updates and a visible `source` per message.

## Design Principles

- **One canonical append-only log** for chat messages (fits how “conversation” behaves and naturally supports syncing).
- **Source is metadata, not role**:
  - `role`: `user | assistant | system`
  - `source`: `ui | cli | sms | geist | reminder | …`
- **Don’t make the supervisor special** to start: use RPC + Redis streams for sync first; only add push/broadcast if needed.
- **Caller-shaped interfaces**: clients should say “append a user message from X” and “tail new messages since Y”.

## Proposed Data Model (Redis Streams)

Use a dedicated Redis Stream per conversation:

- Stream key: `agenda:chat:${conversationId}` (start with `conversationId = "main"`)
- Entry fields (string or JSON-stringified):
  - `type`: `"chat.message"`
  - `role`: `"user" | "assistant" | "system"`
  - `content`: string
  - `source`: `"ui" | "cli" | "sms" | "geist" | ...`
  - `createdAt`: ISO timestamp (still keep Redis stream ID as the canonical ordering)
  - `clientUid`: optional (`AgendaCLI`, browser instance id, sms sender id)
  - `clientMessageId`: UUID generated by the client for dedupe/optimistic UI
  - `meta`: optional JSON (e.g. `{ phone: "+1…", tool: "create_task" }`)

Why streams:
- ordering, persistence, and “sync since last id” are all native.
- messages can be fetched in order without maintaining secondary indexes.

## Service API Shape

Add chat RPCs to `apps/agenda/src/services/database.js` (or extract a dedicated `ConversationService` if `DatabaseService` starts to feel too broad).

### DatabaseService (RPC)

- `appendChatMessage(message)`
  - input: `{ conversationId, role, content, source, clientUid?, clientMessageId?, meta? }`
  - output: normalized message object: `{ id, conversationId, role, content, source, createdAt, clientUid?, clientMessageId?, meta? }`
  - behavior: `XADD agenda:chat:${conversationId} ...`

- `listChatMessages({ conversationId, limit, beforeId? })`
  - output: newest-`limit` messages, ordered oldest→newest (for UI rendering).
  - implement with `XREVRANGE` then reverse (likely requires extending `RedisClient` in `src/db.js` to support reverse range).

- `readChatMessages({ conversationId, afterId, limit })`
  - output: messages strictly after `afterId` (non-blocking).

- `waitForChatMessages({ conversationId, afterId, timeoutMs, limit })`
  - output: either `[]` (timeout) or new messages strictly after `afterId`.
  - implement with `XREAD BLOCK timeoutMs STREAMS key afterId`.
  - this is the “sync primitive” for UI + CLI without changing the supervisor protocol.

### GeistService (RPC)

Replace `interpret(text)` with a caller-shaped method that records the conversation:

- `interpretMessage({ conversationId, text, source, clientUid?, clientMessageId? })`
  - behavior:
    1) `db.appendChatMessage({ role:"user", content:text, source, ... })`
    2) build Claude messages (optionally from recent chat history)
    3) call Claude + tools
    4) `db.appendChatMessage({ role:"assistant", content:assistantText, source:"geist", meta:{ toolsExecuted } })`
    5) return `{ success, response, toolsExecuted, userMessage, assistantMessage }`

Optional (but valuable): make `buildMessages()` pull the last N turns from `DatabaseService.listChatMessages()` so conversation is actually conversational across channels.

## Client Changes

### UI (`apps/agenda/src/ui/app.js`)

- Replace the in-memory “authoritative” chat state with persisted messages:
  - On connect: `listChatMessages({ conversationId:"main", limit:200 })` into `messages()`.
- Add a background sync loop:
  - Maintain `lastSeenId`.
  - Repeatedly call `waitForChatMessages({ afterId:lastSeenId, timeoutMs:20000 })`.
  - Append any returned messages and update `lastSeenId`.
- Update `sendMessage(text)`:
  - Call `GeistService.interpretMessage({ conversationId:"main", text, source:"ui", clientUid, clientMessageId })`.
  - Optionally optimistic-render the user message using `clientMessageId` and replace it when the persisted version arrives.
- Render source:
  - Extend message rendering to show a small badge (e.g. `sms`, `cli`, `ui`, `geist`) alongside role styling.

### CLI (`apps/agenda/bin/agenda.js`)

- Update to call `GeistService.interpretMessage({ source:"cli" })` so CLI participation is persisted.
- Add a “chat mode”:
  - `agenda chat` (or `agenda --interactive --chat`) tails the conversation using `waitForChatMessages` and prints inbound messages with `[source]` tags while still allowing user input.
  - REPL can run a background task that prints new messages as they arrive.

### SMS (new)

Implement an SMS gateway service (under supervisor) that turns inbound SMS into persisted chat messages + Geist calls:

- New service: `apps/agenda/src/services/sms.js` (or `sms_gateway.js`)
  - Connects to supervisor as a node (so it can call `GeistService`/`DatabaseService`).
  - Runs an HTTP server on `AGENDA_HTTP_PORT` (defaults to 3031, per `misc/plans/agenda.md`).
  - Endpoint: `POST /twilio/inbound`
    - verify request (Twilio signature or shared secret)
    - extract `{ from, body }`
    - call `GeistService.interpretMessage({ conversationId:"main", text:body, source:"sms", clientUid:from })`
    - respond with TwiML containing the assistant reply (or send outbound via Twilio API)
- Register this service in `apps/agenda/run.js` as another managed service.

## Dedupe + Ordering

- Treat Redis stream entry id as the canonical message id.
- Use `clientMessageId` for UI/CLI optimistic UI and idempotency:
  - `DatabaseService.appendChatMessage` can keep a short-lived Redis SET like `agenda:chat_dedupe:${conversationId}` to avoid double-appends for the same `clientMessageId`.

## Testing Plan (Simulabra test framework)

  - dedupe with repeated `clientMessageId`
- Add `apps/agenda/tests/integration.js` coverage:
  - calling `GeistService.interpretMessage` produces two persisted messages (user + assistant) with correct `source`.
- Add SMS unit tests:
  - handler parsing + auth failure + happy path using a mocked Geist proxy.

## Rollout Phases

1) **DB primitives**: add stream read/write methods + DatabaseService chat RPCs.
2) **Geist integration**: `interpretMessage` writes both sides of the conversation.
3) **UI sync + source badges**: load history + long-poll tail loop.
4) **CLI sync**: persist CLI inputs; add tail/chat mode.
5) **SMS gateway**: inbound webhook → Geist → TwiML; (optional) outbound reminders.
6) **Hardening**: retention (stream trimming), auth, multi-conversation routing, rate limits.

## Open Questions

- One conversation (`main`) or multiple threads (e.g. per phone number, per project)?
- Should the “Chat” UI also subscribe to `agenda:events` and show task/log/reminder events inline (making it truly “dual-purpose”)?
- What’s the desired behavior when Geist/tools mutate state: should tool execution be logged as `system` chat messages (with `meta`)?
